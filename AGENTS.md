# AI 协作指南

本文件用于约束 AI 在 `ember-docs` 仓库中的协作方式。

## 仓库定位

- 这是 Ember 的公开用户文档站。
- 这里只写普通用户可见、可操作、可验证的使用说明。
- 这里不是内部架构文档仓库，也不是开发规范、部署手册、排障手册仓库。

## 核心原则

- 只写用户视角：只描述用户会看到什么、该点哪里、会发生什么。
- 不编造行为：功能、文案、入口、状态说明必须和实际产品行为一致。
- 不泄露内部细节：不要写内部接口、数据库结构、配置项、部署方式、内部流程。
- 保持直接：文案要短、清楚、能执行，不写空话，不写营销腔。
- 优先改边界：发现内容跨到内部文档范围时，先删错位内容，不要硬补。

## 内容范围

适合放进这个仓库的内容：

- 第一次使用指引
- 播放器推荐
- 用户控制台功能说明
- Telegram Bot 用户用法
- FAQ、更新记录、社群入口

不适合放进这个仓库的内容：

- 后端、前端、Bot 的实现细节
- API、数据库、环境变量、部署步骤
- 内部排障手册、测试手册、迁移方案
- 尚未上线或无法验证的功能规划

## 写作规则

- 所有用户可见内容使用中文。
- 保留必要英文原文：路径、命令、代码标识符、产品名。
- 优先使用“先做什么，再做什么”的顺序表达。
- 一段只讲一件事，避免大段概念解释。
- 如果某个能力受账号状态影响，要明确写清前置条件和限制。
- 如果某个页面或入口名称已经在产品里固定，文档命名必须跟产品一致。

## 修改要求

- 改文档前，先确认改动对应的真实产品行为已经存在。
- 改导航、目录、文案分流时，要同时检查：
  - `README.md`
  - `.vitepress/config.ts`
  - 相关页面间的交叉链接
- 新增页面时，默认要补：
  - 站点导航或侧边栏入口
  - 首页或相关页面的跳转入口
  - 必要的前置条件说明

## 禁止事项

- 不要提交 `node_modules/`
- 不要提交 `.vitepress/dist/`
- 不要提交 `.vitepress/cache/`
- 不要提交 `.DS_Store`
- 不要把主仓库 `ember` 中的内部文档原样搬过来
- 不要把“可能如此”“后续会支持”这类未落地内容写进公开文档

## 验证要求

提交前至少执行：

```bash
npm run docs:build
```

如改动了站点结构或链接，额外建议执行：

```bash
npm run docs:preview
```

构建失败时，不算完成。

## Git 与提交流程

- 不主动提交，除非用户明确要求。
- 提交前先确认 `npm run docs:build` 通过。
- 提交信息使用中文，保持简短明确。

推荐格式：

- `docs: 中文主题`
- `feat(docs): 中文主题`
- `fix(docs): 中文主题`

## 和主仓库的边界

- 产品真实行为仍以主仓库 `ember` 为准。
- 这里负责公开表达，不负责定义内部实现。
- 如果主仓库功能发生用户可见变化，这个仓库需要同步更新。
